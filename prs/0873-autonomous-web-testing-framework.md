# Autonomous Web Testing Framework: AI-Powered QA with browser-use

> **一句话介绍**：让 AI 自己写测试、自己跑测试、自己修测试——基于 browser-use 的全自动化 Web QA 框架，把 QA 团队从重复劳动中解放出来

## 背景

基于 Issue #873，当前 Web 测试面临三大痛点：手动编写测试用例耗时巨大（占开发时间 30-40%）、测试脆弱易碎（UI 改动导致大量用例失效）、维护成本持续攀升（回归测试越积越多）。browser-use（86.5k stars）证明了 AI 操控浏览器的可行性，但尚未有人将其系统性应用于测试领域。

本产品通过 AI Agent 直接理解用户故事、自动生成测试用例、智能执行并自愈修复，打造"测试即服务"的新范式。不是简单的 Selenium 封装，而是真正理解业务逻辑的 AI 测试工程师。

**核心差异化**：不只告诉你"测试失败了"，而是告诉你"为什么失败、如何修复、已自动修复 3 个用例"。

## 产品定位

### 主要竞争对手

| 产品 | 定位 | 优势 | 劣势 | 我们的壁垒 |
|------|------|------|------|-------------|
| **Selenium** | 浏览器自动化 | 成熟稳定，生态丰富 | 需要手写代码，维护成本高 | AI 生成 + 自愈，无需手写 |
| **Cypress** | 现代 E2E 测试 | 开发者友好，实时重载 | 仅支持特定浏览器，学习曲线 | 跨浏览器 + AI 智能定位 |
| **Playwright** | 跨浏览器测试 | 微软背书，多语言支持 | 仍需手动编写和维护 | AI 全自动生成和维护 |
| **Mabl** | AI 测试平台 | 无代码，自愈能力 | 价格昂贵（$500+/月），功能受限 | 开源 + 更深层次的 AI 理解 |
| **Testim** | AI 驱动测试 | 智能定位，视觉 AI | 闭源，定制能力弱 | 完全开源，可深度定制 |

### 核心护城河
1. **深度语义理解**：不只识别元素位置，理解业务含义（"登录按钮" vs "蓝色矩形"）
2. **自愈定位系统**：UI 改动后自动寻找替代选择器，测试不会因 UI 微调而失效
3. **AI 生成 + 人工审核**：生成高质量测试，人工一键确认，效率提升 10 倍
4. **视觉回归检测**：结合截图对比和 AI 理解，发现人类容易忽略的视觉差异
5. **持续学习**：每次修复都训练模型，越用越准确

## 技术架构

### 整体架构
```
用户故事/PRD 输入
        ↓
  LLM 场景理解 + 测试用例生成（GLM-4 / Claude）
        ↓
  测试用例 → 结构化测试计划（JSON/YAML）
        ↓
  browser-use 引擎 → 浏览器自动化执行
        ↓
  多策略元素定位（CSS → XPath → 视觉 → AI 语义）
        ↓
  执行结果收集 → 视觉截图 + DOM 快照 + 网络日志
        ↓
  AI 分析引擎 → 通过/失败判定 + 根因分析 + 自愈修复
        ↓
  测试报告 → CI/CD 集成 + 仪表盘展示 + 通知推送
```

### 关键技术细节

**多策略元素定位引擎**：
```python
class SmartLocator:
    """四级定位策略，自动降级"""
    
    def locate(self, element_description: str, context: PageContext) -> Element:
        # Level 1: CSS 选择器（最快）
        element = self.css_selector.locate(element_description, context)
        if element: return element
        
        # Level 2: XPath（更灵活）
        element = self.xpath.locate(element_description, context)
        if element: return element
        
        # Level 3: 视觉匹配（基于截图）
        element = self.visual_matching.locate(element_description, context)
        if element: return element
        
        # Level 4: AI 语义理解（最强大，最慢）
        element = self.ai_semantic.locate(element_description, context)
        return element
```

**自愈测试流程**：
```
测试执行 → 元素未找到 → 触发自愈流程
    ↓
分析元素历史定位记录（之前用什么选择器找到的）
    ↓
DOM Diff 分析（页面结构发生了什么变化）
    ↓
多策略重新定位（CSS → XPath → 视觉 → AI）
    ↓
验证新元素（标签名、角色、可见性、可交互性）
    ↓
成功 → 更新测试用例 + 记录修复模式
失败 → 报告 + 人工审核队列
```

**测试生成 Prompt 模板**（约 2500 tokens）：
```
你是一位资深的 QA 工程师，请根据以下用户故事生成完整的测试套件。

用户故事：{user_story}

技术栈信息：{tech_stack}
已有路由：{routes}
已有组件：{components}

请生成以下内容（JSON 格式）：
1. 测试用例列表（每个包含：名称、前置条件、步骤、预期结果、优先级）
2. 边界条件测试（输入验证、极端值、并发场景）
3. 负面测试（错误处理、异常流程）
4. 数据驱动测试数据集

每个步骤包含：
- action: 用户操作描述
- target: 目标元素描述（语义化）
- expected: 预期结果
```

**视觉回归检测算法**：
```
截图 A（基线）vs 截图 B（当前）
    ↓
SSIM 结构相似度计算（快速筛选）
    ↓
差异区域检测（bounding box 定位）
    ↓
AI 语义判断（这个差异是有意义的 UI 变更还是无关抖动？）
    ↓
分类：✅ 忽略（动画/时间戳）| ⚠️ 确认（布局偏移）| ❌ 缺陷（元素缺失）
```

## 使用示例

### CLI 快速开始
```bash
# 安装
pip install autotest-ai

# 初始化项目
autotest init --project my-webapp

# 从用户故事生成测试
autotest generate --story "用户登录：输入用户名和密码，点击登录按钮，成功后跳转到首页"

# 执行测试
autotest run --browser chrome --headless

# 查看报告
autotest report --format html --open
```

```

```

## 技术对比详解

### vs Selenium（传统方案）
| 维度 | Selenium | Autotest AI |
|------|----------|-------------|
| 测试编写 | 手动编码（Java/Python） | AI 自动生成 |
| 维护成本 | 高（UI 改动 = 大量修改） | 低（自愈引擎自动修复） |
| 学习曲线 | 陡峭（需要编程 + 定位知识） | 平缓（自然语言描述即可） |
| 执行速度 | 中等 | 快（并行 + 智能调度） |
| 视觉测试 | 不支持 | AI 视觉回归检测 |
| 失败分析 | 人工排查 | AI 根因分析 |

### vs Mabl（商业 AI 测试）
| 维度 | Mabl | Autotest AI |
|------|------|-------------|
| 定价 | $500+/月（起步） | 免费（社区版）/ ¥299/月（Pro） |
| 部署方式 | 仅 SaaS | SaaS + 私有化部署 |
| 定制能力 | 有限 | 完全开源可定制 |
| LLM 集成 | 基础 ML | 最新 LLM（GLM-4/Claude） |
| 社区 | 闭源 | 开源社区驱动 |
