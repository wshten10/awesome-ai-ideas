# AI 故事世界共创师

> **一句话卖点**：下班后躺在沙发上的职场人，用 AI 10 秒进入角色扮演世界，从被动刷 2 小时视频看完就忘，到主动创造属于自己的故事世界

## 概述

基于 Issue #149 创建的详细设计文档。

现代人的娱乐方式陷入「被动消费」陷阱：刷 2 小时短视频，笑完就忘；想玩桌游 RPG 找不到队友；想体验剧情游戏受限于固定脚本。结果是：娱乐时间过去了，但什么都没留下，反而更空虚。

本产品通过自然语言角色扮演 + 动态世界生成 + 多角色互动，让用户从「消费故事」变成「创造故事」，每次冒险都是独一无二的体验。

## 功能设计

### 核心功能

1. **自然语言角色扮演**
   - 直接对话，无需学习规则
   - AI 理解用户意图并推进剧情
   - 支持任意角色设定（末世机械师、武侠剑客、星际船长...）

2. **动态世界生成**
   - 用户说「我想当海盗」→ 10 秒进入海盗船场景
   - AI 根据用户选择实时调整世界观和剧情走向
   - 每次选择都有蝴蝶效应，故事永不重复

3. **多角色互动**
   - AI 扮演多个 NPC，每个都有独特性格和背景
   - NPC 之间有互动，世界有生命力
   - 支持「剧情导演」模式，协调 NPC 行为避免矛盾

4. **故事保存分享**
   - 完成的冒险保存为「故事书」
   - 支持导出 Markdown/JSON 格式
   - 朋友可以续写你的故事

5. **多种题材**
   - 内置题材：末世生存、武侠江湖、科幻星际、悬疑推理、恋爱养成
   - 支持自定义世界观

6. **「随时暂停」机制**
   - 现实中有事可以随时暂停
   - 回来时「前情提要」快速回忆

### 差异化功能

| 现有方案 | 本产品 |
|---------|--------|
| 刷视频 2 小时，看完就忘 | 创作自己的故事，每次都不同 |
| 想玩桌游找不到人 | AI 24 小时在线，随时开始 |
| 剧情游戏固定分支 | AI 实时生成，无限可能 |
| 只能消费不能创造 | 边玩边创，故事属于你 |

## 实现步骤

### Phase 1：MVP 验证（3-4 周）

**目标**：验证「AI 故事共创」的 PMF，单一题材深度打磨 + 多 Agent 核心验证

**⚠️ 时间风险预警**：
- 3-4 周完成 MVP + 双 NPC 互动验证，时间紧张
- **风险缓解**：
  1. 优先保证核心对话质量，双 NPC 互动可简化（仅 2 个 NPC，1 个场景）
  2. 如第 3 周进度落后，延迟到第 5 周，不影响整体节奏
  3. 提前准备 Plan B：先上线单 NPC 版本，双 NPC 作为后续更新

**第 1 周：基础设施**
- 后端架构搭建（Node.js + PostgreSQL + Redis）
- LLM 集成（Claude/GPT-4 + 本地模型）
- 基础对话接口

**第 2 周：核心对话**
- 单一题材：**末世生存**（中国用户接受度高）
- 基础角色扮演对话
- 简单分支选择（2-3 个选项）
- 状态管理（角色属性、物品）

**第 3 周：世界生成 + 双 NPC 互动验证**
- 3-5 个预设场景（废墟、安全屋、荒野、城镇、避难所）
- 2-3 个 NPC（医生、工程师、商人）
- **核心验证：简单双 NPC 互动**（医生和工程师同时在线，有基础对话）
- 简单剧情线（3-5 个关键节点）

**第 4 周：用户验证**
- 邀请 100 名种子用户（见下文获取渠道）
- 数据埋点（对话轮次、留存、NPS）
- 快速迭代修复 bug

**成功标准**：
- 7 日留存 > 25%
- 平均对话轮次 > 20 轮（提升标准，与「完整故事体验」定位对齐）
- 故事完成率 > 40%（新增指标：完成主线剧情的用户比例）
- 用户 NPS > 30

## 资源需求

### API

- **LLM API**：GPT-4 / Claude / kimi-k2.5
  - 用途：对话生成、场景生成、剧情推进
  - 预估成本：¥0.05-0.15/对话轮次
  - 优化方案：本地小模型 + 云端大模型混合

**LLM 混合使用策略**：

| 场景 | 模型 | 原因 |
|------|------|------|
| **输入初筛** | 本地模型（Qwen-7B） | 快速、低成本，过滤敏感词 |
| **简单对话** | 本地模型 | 成本低，延迟小 |
| **剧情生成** | 云端模型（Claude/GPT-4） | 质量高，保持一致性 |
| **关键节点** | 云端模型 | 重要决策，不能出错 |
| **NPC 日常对话** | 本地模型 | 批量生成，成本敏感 |

**具体分工示例**：
- 用户：「我想和医生聊聊」→ 本地模型（简单 NPC 对话）
- 用户：「我决定炸掉避难所」→ 云端模型（关键剧情节点）
- 用户：「继续探索」→ 本地模型（常规推进）
- 用户：「医生和工程师吵架了」→ 云端模型（多 Agent 协调）

**成本优化效果**：70% 请求走本地模型，总成本降低 50%

**模型质量验证（Phase 1 A/B 测试）**：
| 场景 | A 组（纯云端） | B 组（混合） | 验证指标 |
|------|---------------|-------------|----------|
| NPC 日常对话 | GPT-4 | Qwen-7B | 用户满意度、对话轮次 |
| 剧情生成 | Claude | Claude | -（对照组） |
| 关键节点 | GPT-4 | GPT-4 | -（相同） |

**验证周期**：Phase 1 第 3-4 周
**成功标准**：混合组用户满意度不低于纯云端组 90%，成本降低 > 40%
**失败标准**（放弃混合方案，回归纯云端）：
- 混合组用户满意度低于纯云端组 80%
- 故事一致性投诉率 > 10%（对照组 < 3%）
- 用户主动流失率 > 20%（对照组 < 15%）
- A/B 测试第 2 周仍无明显改善

- **语音 API**（可选）：ElevenLabs / Azure TTS
  - 用途：AI 语音演绎
  - 预估成本：¥0.02/秒

### 服务

- **后端**：Node.js / Python
- **数据库**：PostgreSQL（故事状态） + Redis（会话缓存）
- **存储**：S3（故事书导出）
- **实时通信**：WebSocket（对话流）

### 开发团队

- **MVP 阶段**：2 人（1 后端 + 1 前端）
- **扩展阶段**：3-4 人（+1 Prompt 工程师）

### LLM 成本（可变）

- 平均每用户每月：20 轮对话 × ¥0.10 = ¥2.0
- 订阅用户：50 轮 × ¥0.10 = ¥5.0
- 混合成本（5% 付费率）：¥2.35/用户/月

**成本优化策略（对话压缩）**：
- **问题**：长对话 token 累积，成本可能超预期
- **方案 1：滑动窗口**：只保留最近 20 轮对话作为上下文，更早内容压缩为摘要
  - **实现**：每 5 轮调用一次 LLM 生成「剧情摘要」（约 100 tokens），替代原始对话（约 500 tokens）
  - **一致性保障**：摘要包含关键决策、角色状态变化、重要物品
- **方案 2：关键事件提取**：每 10 轮提取「关键事件摘要」（50 tokens），替代完整对话（500+ tokens）
  - **技术实现**：使用本地模型（Qwen-7B）+ 规则引擎
  - **提取规则**：用户决策、NPC 行为变化、物品获取/丢失、关系变化
- **方案 3：成本上限**：免费用户每月最多 60 轮，订阅用户 200 轮，超出需额外付费
- **预估效果**：成本降低 40-60%，¥0.04-0.06/轮次
- **验证计划**：Phase 1 对比三种方案的实际效果（成本、一致性、用户体验）

### 单位经济分析

| 指标 | 免费用户 | 订阅用户 |
|------|----------|----------|
| LLM 成本 | ¥2.0/月 | ¥5.0/月 |
| 分摊固定成本 | ¥0.23/月 | ¥0.23/月 |
| **总成本** | **¥2.23/月** | **¥5.23/月** |
| 收入 | ¥0 | ¥19.9/月 |
| **毛利** | **-¥2.23** | **¥14.67** |

**盈亏平衡点**：每 1 个付费用户需补贴 6.6 个免费用户（14.67 / 2.23 ≈ 6.6）

**LTV 计算**：
- 订阅用户平均留存：3 个月
- LTV = ¥19.9 × 3 = ¥59.7
- LTV/CAC = 59.7 / 25 = **2.39**（健康区间 2-4）

**结论**：¥19.9/月 定价合理，但需控制免费用户成本（限制轮次 + 本地模型）

## 内容安全方案

### 三层防护体系

**第 1 层：输入过滤（用户发送前）**
- **技术方案**：阿里云/腾讯云内容安全 API
- **检测内容**：
  - 敏感词（政治、暴力、色情）
  - 恶意 Prompt（越狱、角色扮演绕过）
- **处理方式**：拦截 + 提示「请修改内容」
- **成本**：¥0.002/次，月均 ¥500（1 万用户 × 25 次）

**第 2 层：输出过滤（AI 生成后）**
- **技术方案**：同上 API + 自研规则引擎
- **检测内容**：
  - AI 生成的敏感内容
  - 剧情崩坏（逻辑矛盾、价值观问题）
- **处理方式**：重新生成 + 记录日志
- **优化**：本地小模型预过滤，降低 API 调用

**第 3 层：人工审核（事后抽查）**
- **审核对象**：
  - 用户举报的故事
  - 高频互动故事（>50 轮对话）
  - 随机抽样（5% 新故事）
- **审核标准**：
  - 不涉及政治敏感
  - 不宣扬暴力/歧视
  - 符合社会主义核心价值观
- **处理流程**：
  - 违规故事：下架 + 警告用户
  - 严重违规：封禁账号
- **人力成本**：1 名兼职审核，¥3000/月

### 规模化审核策略（MAU > 5000）

**问题**：人工审核无法随用户增长线性扩展

**解决方案**：
1. **AI 辅助审核**：
   - 训练轻量级分类模型（基于用户举报数据）
   - 高风险内容自动标记，人工复核
   - 预估成本：¥1000/月（API 调用）

2. **用户举报优先**：
   - 建立举报激励机制（举报违规得积分）
   - 优先处理举报内容，降低抽查比例
   - 人力聚焦高风险内容

3. **分等级审核**：
   - 新用户：5% 抽查
   - 活跃用户（>10 个故事）：2% 抽查
   - 优质创作者（高评分、无举报）：免抽查
   - 有举报记录：10% 抽查

4. **成本预估（MAU 10000）**：
   - 人工审核：2 名兼职，¥6000/月
   - AI 辅助：¥1000/月
   - 总计：¥7000/月（vs 固定成本 ¥3000/月）

### 合规策略

- **备案**：ICP 备案 + 网络文化经营许可证
- **未成年人保护**：实名认证 + 时间限制
- **数据安全**：用户数据加密存储，7 天后删除对话记录

## 多 Agent 协调机制

### 架构设计

```
┌─────────────┐
│  剧情导演   │ ← 协调中心
└──────┬──────┘
       │
   ┌───┴───┬───────┬───────┐
   │       │       │       |
┌──▼──┐ ┌──▼──┐ ┌──▼──┐ ┌──▼──┐
│ NPC1│ │ NPC2│ │ NPC3│ │ 场景│
└─────┘ └─────┘ └─────┘ └─────┘
```

### 剧情导演职责

1. **一致性维护**
   - 记录关键事件（用户选择、NPC 行为、世界观变化）
   - 检查新内容是否与历史冲突
   - 冲突时：提示 NPC 修改或重新生成

2. **优先级规则**
   - **用户意图 > NPC 个性**：用户说「我想和医生合作」，医生不能无理由拒绝
   - **剧情推进 > NPC 自由**：NPC 行为需服务主线，不能跑题
   - **安全合规 > 一切**：任何内容不得违反安全规则

3. **冲突解决**
   - **NPC 间冲突**：剧情导演协调，让一方妥协或创造新剧情
   - **用户 vs NPC 冲突**：尊重用户选择，NPC 适应
   - **逻辑矛盾**：回溯最近 3 轮对话，找出矛盾点并修正

### 协调示例

**场景**：用户选择「攻击商人」，NPC 医生之前说「商人是好人」

- 剧情导演检测到冲突
- 协调方案：
  1. 医生改口「其实商人也有阴暗面」
  2. 或医生表示「我不认同你的做法」→ 增加角色深度
  3. 优先选择方案 2（更真实）

### 通信协议

**架构选择**：事件驱动 + 消息队列

```
用户输入 → 剧情导演（事件总线）
              ↓
        ┌─────┼─────┐
        ↓     ↓     ↓
      NPC1  NPC2  NPC3
        ↓     ↓     ↓
        └─────┼─────┘
              ↓
        剧情导演（合并响应）
              ↓
           用户输出
```

**通信机制**：
1. **同步场景**（用户直接与 NPC 对话）
   - 剧情导演 → 目标 NPC → 直接响应
   - 延迟：< 2 秒

2. **异步场景**（多 NPC 同时在线）
   - 剧情导演广播事件 → 所有相关 NPC 生成响应
   - 剧情导演按优先级合并 → 输出给用户
   - 延迟：< 5 秒（并行处理）

3. **冲突检测**（每次 NPC 响应前）
   - 剧情导演检查历史状态
   - 冲突时：重新生成或调整
   - 最大重试次数：3 次

### 延迟控制方案（目标：< 2 秒）

**问题**：多 Agent 协调可能引入额外延迟

**解决方案**：

1. **并行处理 + 流式响应**
   - 所有 NPC 同时生成响应（并行调用 LLM）
   - 流式返回第一个完成的 NPC 响应
   - 其他 NPC 响应作为后续补充

2. **缓存策略**
   - NPC 日常对话缓存（相似场景复用）
   - 场景描述缓存（相同地点）
   - 缓存命中率预估：30-40%

3. **本地模型加速**
   - Qwen-7B 本地部署，延迟 < 500ms
   - 简单对话优先走本地模型
   - 云端模型仅用于关键剧情

4. **降级方案**
   - P95 延迟 > 3 秒时：自动降级到单 NPC 模式
   - 高峰期：暂停多 NPC 互动，恢复单 NPC 体验
   - 监控指标：实时延迟 P50/P95/P99

**延迟预估**：
| 场景 | 模型 | 预估延迟 |
|------|------|----------|
| 单 NPC 日常对话 | 本地模型 | 500-800ms |
| 单 NPC 关键剧情 | 云端模型 | 1-2s |
| 多 NPC 互动 | 并行云端 | 2-3s |
| 多 NPC + 缓存命中 | 混合 | 1-1.5s |
